# ServiceCraft — Функциональность

## Архитектура системы

Система состоит из двух функциональных слоёв:

1. **HiTL KB Pipeline** — реализован, соответствует диплому
2. **Legacy Audio Pipeline** — реализован ранее, описан в дипломе как «предшествующий подход»

---

## Слой 1: Human-in-the-Loop Knowledge Base (дипломная работа)

### Откуда берутся вопросы

798 уникальных вопросов импортированы из `knowledgeos-export.json` — реальные вопросы клиентов, извлечённые из звонков, семантически дедуплицированные и ранжированные по частоте встречаемости.

**Модель `Question` (PostgreSQL):**
- `question` — текст вопроса
- `answer` — ответ оператора (пустой → статус `unscripted`)
- `ai_answer` / `ai_status` — черновик от LLM + статус (pending / accepted / edited)
- `theme_id` / `theme_name` — тематическая группа
- `frequency` — сколько звонков содержат этот вопрос
- `rag_approved` / `indexed_at` — флаг публикации в RAG и время индексации
- `evidence` — JSONB со ссылками на звонки-источники

---

### Страница «Очередь» `/questions`

**Что делает:** показывает вопросы без ответа (`status = unscripted`), отсортированные по частоте.

**Поток скриптования оператора:**
1. Оператор видит вопрос + частоту упоминания в звонках
2. Нажимает «Сгенерировать черновик» → LLM предлагает ответ на основе FAQ-контекста
3. Оператор редактирует черновик или пишет с нуля
4. Нажимает «Утвердить → Q&A»:
   - Сохраняется `answer`, устанавливается `rag_approved = true`, `indexed_at = now`
   - **Синхронно** пара индексируется в Qdrant коллекцию `qa_pairs`
   - Пара перемещается в раздел «База Q&A»

**Дополнительно:**
- Кнопка «В RAG» (bulk) — переиндексирует все scripted-вопросы (`POST /questions/reindex`)
- Импорт/экспорт JSON
- Фильтры по теме и полнотекстовый поиск

---

### Страница «База Q&A» `/qa`

**Что делает:** показывает все отвеченные пары (`status = scripted`).

- Редактирование ответа прямо на странице (inline textarea)
- Кнопка «Утвердить изменения» → сохраняет + переиндексирует в Qdrant
- Кнопка «Утвердить в RAG» → переиндексирует без изменений
- Бейдж «В RAG» с датой индексации (tooltip)

---

### Гибридный RAG (Qdrant `qa_pairs`)

**Как устроен:**

```
Запрос пользователя
    ↓
OpenAI text-embedding-3-large (dense, 1024d, cosine)
BM25 Sparse Encoder (vocab 2826 терминов, k1=1.2, b=0.75, Robertson IDF)
    ↓
Qdrant qa_pairs — named vectors: dense + sparse (inverted_index)
Prefetch: dense top-20 + sparse top-20 (если не всё стоп-слова)
    ↓
RRF Fusion (k=60) → top-5 результатов
    ↓
FAQ контекст (статические записи клиники: адреса, часы, политики)
    ↓
YandexGPT Pro — ответ с контекстом
```

**Контекст агента строится из:**
1. `═══ FAQ (Данные клиники) ═══` — адреса, часы, врачи, правила приёма (из таблицы `faqs`)
2. `═══ БАЗА ЗНАНИЙ (Q&A) ═══` — top-5 релевантных пар из `qa_pairs`
3. `═══ ПРАЙС ═══` — цены по ключевым словам (если релевантно)
4. YClients контекст — если обнаружен интент на запись

**Что НЕ входит в RAG:** статьи из базы знаний (`articles`) — они созданы Legacy Pipeline и агентом не используются.

### Голосовой агент `/agent`

**LiveKit WebRTC голосовой звонок.**

```
Браузер (микрофон) → LiveKit Room → Python агент (LiveKit Agents SDK)
    → STT: gpt-4o-transcribe (Replicate)
    → RAG: function call → /api/agent/context → hybrid Q&A search (Go)
    → LLM: YandexGPT Pro (streaming)
    → TTS: OpenAI TTS (nova/shimmer/echo)
    → Браузер (аудио)
```

Python агент находится отдельно от Go-бэкенда. Go обслуживает `/api/agent/context` — возвращает RAG-контекст по слагам.

---

### Чат-агент `/agent` (chat tab)

Текстовый чат. Тот же Go AgentService: гибридный поиск → FAQ → YandexGPT. История диалога хранится в таблице `chat_messages`.

---

### Настройки Lens `/settings`

- **Confidence threshold** — слайдер 0.0–1.0 (хранится в `company.settings`, не влияет на runtime агента)
- **YClients API key** — поле для ввода + кнопка «Test Connection» (mock)
- **TTS voice** — выбор голоса (nova / shimmer / echo)
- **Каналы** — переключатели Voice / Chat

---

### YClients (mock) `/api/yclients/*`

Симулирует интеграцию с МИС YClients:
- `GET /slots` — доступные слоты на 7 дней (реалистичные mock-данные)
- `POST /book` — запись на приём
- `GET /patients/:phone` — информация о питомце по телефону

Интент на запись детектируется в агенте по ключевым словам → контекст слотов инжектируется в промпт.

---

### Аналитика `/analytics`

- **KPI карточки:** всего вопросов, отвечено (% покрытие), без ответа, LLM acceptance rate
- **Прогресс-бар** покрытия базы знаний (данные из `/api/questions/stats`)
- **RAG метрики** (тестовая выборка 50 вопросов): BLEU 0.41, ROUGE-L 0.58, BERTScore 0.81
- **Архитектурная карточка** Hybrid RAG: Query → dense + sparse → Qdrant → RRF → YandexGPT
- **Топ-10 вопросов** по частоте с статусом и RAG-бейджем

---

### FAQ `/faq`

Ручное редактирование статических записей клиники (адреса, часы, политики, врачи, часто задаваемые вопросы). Приоритет в RAG выше, чем Q&A.

## Слой 2: Legacy Audio Pipeline («предшествующий подход»)

В дипломе этот подход описан как **неудавшийся** и замещённый HiTL-пайплайном. В коде сохранён, работает независимо от HiTL.

### Страница «Наполнение» `/pipeline`

**Что делает:** принимает аудиофайлы звонков и автоматически создаёт/обогащает статьи базы знаний.

**Поток обработки одного файла:**

```
Аудиофайл (mp3/wav/ogg, до 500 МБ)
    ↓
Stage 1 — Транскрипция
  gpt-4o-transcribe (через Replicate API)
  SHA256-кэш в таблице transcription_cache — повторная загрузка → мгновенно
  Результат: размеченный транскрипт [Оператор]: / [Клиент]:
    ↓
Stage 2 — Сегментация
  LLM (YandexGPT) разбивает транскрипт на тематические сегменты
  Каждый сегмент: { topic, text, urgency, category, suggested_slug, suggested_name }
    ↓
Stage 3 — Классификация (Qdrant коллекция `articles`)
  Эмбеддинг сегмента → косинусная близость к существующим статьям
  Порог 0.50: выше → обогащение существующей, ниже → создание новой статьи
    ↓
Stage 4 — Обогащение / Создание статьи
  Если статья найдена: YandexGPT дополняет содержимое новыми фактами из сегмента
  Если не найдена: создаётся новая статья (slug, category, content)
  Параллельная обработка сегментов с per-slug mutex
```

**Batch режим:** до 20 файлов, 5 параллельных воркеров.

**Результат на экране:** транскрипт + таблица сегментов (тема, urgency, действие, slug, similarity score).

---

### База знаний — статьи `/articles`

Статьи, созданные Legacy Pipeline. Каждая имеет: slug, название, категорию, JSONB content, эмбеддинг в Qdrant (`articles`, 1024d cosine).

**В RAG агента НЕ используются.** Остаются как редактируемая wiki и данные для кластеризации.

---

### Карта кластеров `/clusters`

Визуализация семантических кластеров статей (force-directed graph на D3.js).

### Граф диалогов `/graph`

Визуализация типичных сценариев диалогов как графа переходов между темами.

### Прайс-дерево `/price-tree`

Иерархическое дерево услуг и цен клиники. Используется агентом для ценовых запросов.

## Схема данных

### PostgreSQL

| Таблица | Назначение | Слой |
|---------|-----------|------|
| `questions` | 798 Q&A пар, очередь скриптования | HiTL |
| `faqs` | Статические данные клиники | HiTL |
| `chat_messages` | История чата | HiTL |
| `articles` | Статьи базы знаний | Legacy |
| `transcription_cache` | SHA256 кэш транскрипций | Legacy |
| `companies` | Мультитенантность | Общее |
| `users` | Аутентификация (JWT HS256) | Общее |
| `price_items` | Прайс-лист | Общее |

### Qdrant (векторная БД)

| Коллекция | Назначение | Векторы | Слой |
|-----------|-----------|---------|------|
| `articles` | Эмбеддинги статей | dense 1024d cosine | Legacy |
| `qa_pairs` | Гибридный поиск Q&A | dense 1024d + sparse BM25 | HiTL |
